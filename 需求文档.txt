项目名称：EduMatch（Demo）
文档版本：2026-04-09（含对话整理）
文档用途：当前实现需求基线（产品 + 交互 + 视觉）

一、产品定位
1) 演示型留学匹配产品，目标是“流程完整、操作可点、展示清晰”。
2) 重点不是后端能力，而是端到端体验闭环：
   首页 -> 问卷 -> 匹配结果 -> 工作台。
3) Demo 优先级高于严格校验，避免关键路径阻塞。

二、用户与数据
1) 用户角色：游客（无登录）。
2) 数据存储：浏览器本地存储（localStorage）。
3) 当前不包含：账号系统、云端同步、真实文件上传、真实 AI API。

三、页面与流程需求
1) 首页（/）
   - 进入问卷主入口。

2) 问卷页（/questionnaire）
   - 左侧步骤导航 + 右侧滚动表单。
   - 滚动时步骤自动高亮，点击步骤可跳转对应段。
   - 顶部右侧操作使用 icon：
     a. 上传材料（图标按钮）
     b. 去匹配（图标按钮）
   - 底部固定“下一步/完成”按钮，支持流程推进。

3) 匹配页（/match）
   - 自动生成匹配结果（本地规则 + 模拟延迟）。
   - 左侧学校列表，右侧项目列表；分类筛选：全部 / 冲刺 / 主申 / 保底。
   - 学校卡右侧提供「加院校」图标按钮：一键将该校在当前匹配结果中的全部项目写入申请单（edumatch_added_programs）；已全部加入时变为勾选态，再次点击可一键移除此校全部已加项目；部分加入时显示「部分已加」并仍可用加号补全其余项目。
   - 可添加/移除项目到申请单；底部固定栏展示已选数量与去工作台入口。
   - 信息层级（2026-04-09）
     a. 大屏（lg+）：根布局使用视口高度（dvh），页面主体不随列表无限增高；左右两栏各自内部滚动，尽量一屏内看到顶栏 + 筛选 + 双栏内容。
     b. 顶栏压缩高度；筛选 Tab 与「重匹配」图标按钮同一工具行。
     c. 去掉与列表重复的统计大标题（如顶部的「X 校 · X 项」）；去掉「学校 / 项目」等分栏标题行，由列表本身表达结构。
     d. 顶栏仅保留品牌区与操作（返回问卷、工作台）；不写「匹配结果」面包屑式文案。
     e. 侧栏「全部」入口：网格图标 + 当前筛选下的项目数量；样式与学校卡选中态一致。
     f. 已选项目在列表内有边框/底色区分；底部栏文案精简（如「已选 n」）。
   - 文书：左侧点选学校后在右侧项目卡可进入文书页。已有非空草稿时显示「文书」；已加入申请单但尚无草稿时显示「写文书」。学校卡上展示该校下有多少个项目已有文书草稿（与 localStorage 同步，窗口 focus/visibility 时刷新）。
   - 文书页访问规则与工作台对齐：若项目未加入工作台但本地已有该项目的草稿，仍可从匹配页「文书」进入编辑。

4) 工作台（/workspace）
   - 布局参考 Notion：左侧固定侧栏 + 右侧主内容；小屏侧栏收入抽屉，顶栏菜单打开。
   - 侧栏：搜索、浏览（全部申请 / 即将截止）、按学校筛选、进度摘要、快捷链（匹配 / 我的背景，链至问卷页）。
   - 主区：标题随筛选变化、添加项目、按学校分组列表。
   - 更新申请状态；截止提醒（全部视图下展示摘要）。
   - 列表默认信息精简，hover 显示补充信息；状态标签使用中性色。
   - 文书：每个已加项目提供「写文书」入口，跳转 /workspace/write/[programId]；支持个人陈述、简历大纲、Why School、推荐信提纲四类 Tab，按学校与项目预填提纲；草稿 debounce 写入 localStorage（edumatch_document_drafts）；工作台展示「草稿」提示；从文书页返回工作台后 focus/visibility 时刷新草稿状态。
   - 侧栏「我的文书」列出所有已有非空草稿的项目（按最近更新时间），点击进入对应文书页；匹配结果中已无该项目时标题显示为截断的草稿标识。

四、问卷模块需求（当前实现）
1) 个人信息（极简）
   - 字段：姓名、目标专业、学位、本科专业、申请领域、入学时间、预算、学制打算、地区。
   - 目标专业：希望申请的具体项目/专业方向；申请领域：更宽的一级方向（可与目标专业互补用于匹配）。
   - 预算、学制打算为自由文本，便于 Demo 快速填写。
   - 地区支持两种视图：列表 / 地图。
   - 文案简化，辅助提示最小化。
2) 教育背景
   - 多条经历增删改；匹配说明会优先引用首条教育经历的学校与专业（与「本科专业」字段互补）。
3) 标化成绩
   - TOEFL / IELTS / GRE / GMAT 输入。
   - 已移除页面底部提示文案。
4) 工作经历
   - 公司名称/职位输入框已去掉示例提示。
5) 技能
   - 信息层级重排：已添加技能优先展示，添加区其次，推荐区最后。
6) 上传材料
   - 顶部入口触发面板。
   - 当前仅本地元信息，不做云端上传。

五、匹配逻辑需求（当前实现）
1) 输入维度
   - 国家偏好、目标专业与申请领域及本科专业（合并参与关键词匹配）、目标学位、预算与学制（写入匹配原因、学制数字接近时略加权）、教育与经历信息。
2) 结果规模
   - 学校约 8 所；项目 12-15 个。
3) 规则
   - 国家优先筛选，不足自动补齐。
   - 专业关键词中英匹配。
   - 学位层级匹配。
   - 生成匹配原因列表，供项目卡首条「匹配说明」展示。
4) 匹配文案原则（2026-04-09）
   - 不在匹配原因中出现「建议申请批次」类话术；入学学期仍在问卷中收集（用于流程完整性），但不作为匹配结果里的批次建议输出。
   - 首条匹配原因应基于问卷已填基础信息动态拼接，例如：教育经历首条的「学校 · 专业」、个人信息中的「本科专业」兜底、以及「目标专业 / 申请领域」；文案需与填写内容一致，不臆造背景。
   - 无匹配原因时的兜底句：强调「结合问卷基础信息与项目方向」，不出现「申请节奏」等批次导向提示。
5) Demo 策略
   - 保证始终可生成可展示结果。

六、视觉与信息架构规范（当前生效）
1) 信息去重
   - 同一信息在同页只保留一处关键展示。
   - 示例：已加项目数不在顶部重复显示。
2) 层级策略
   - 默认显示核心信息；次要信息折叠或按需展开。
3) 色彩策略
   - 降低颜色数量与面积，采用中性配色为主；选中态可用主色弱描边/弱 ring。
4) 阴影策略
   - 全局 UI 保持干净；匹配页列表卡片允许轻阴影（shadow-sm）与 hover 略增强，用于区分卡片与背景层级。
5) 文案策略
   - 能短则短；能用图标表达的标签尽量不用文字（如学制、截止、费用、排名、项目数等），数值与图标并列即可。
6) 无障碍
   - 图标按钮与纯图标信息条需配置 aria-label / aria-hidden，避免读屏缺失语义。

七、匹配页卡片规范（2026-04-09，与实现对齐；2026-04-09 修订：基础信息与课程摘要）
1) 学校卡
   - 主标题：英文校名（nameEn）；副标题：中文校名（name）。
   - 第三行：城市 · 国家（地图钉图标）；档位标签：冲刺 / 主申 / 保底（Badge）。
   - 第四行：奖杯 + 排名数字；列表图标 + 项目数量；若有文书草稿则文书图标 + 数量；若该校项目仅部分在申请单中则显示「部分已加」。
   - 右侧竖条：加院校按钮（Plus / 已全时 Check），与左侧「选中该校、筛选右侧列表」分区；加院校操作写入与项目卡相同的本地申请单。
   - 圆形校徽/首字母；圆角卡片、轻阴影；选中态边框/底色/弱 ring。
   - 左侧主区为选中按钮，aria-label 含中英校名、城市、国家、排名、项目数；右侧按钮独立 aria-label。

2) 项目卡
   - 标题区：英文项目名（nameEn）；中文项目名（name）；院系（department）；学位小标签；侧栏已选中某校时头部不重复该校英文名。
   - 信息格（默认可见）：2×2（小屏）/ 1×4（sm+）网格——学制、截止、学费参考、申请费；仅图标 + 数值/缩写，具体含义用 aria-label。
   - 项目简介：description 默认展示约两行（line-clamp），展开后全文。
   - 匹配说明行：星形图标 + matchReasons[0]。
   - 展开区顺序：简介（文档图标）→ 课程与培养（书本图标，数据字段 curriculumNote，由 mock 按项目类型生成）→ 申请要求（清单图标）。申请费已在信息格展示，展开区不再重复。
   - 添加 / 已加、文书入口、校徽头像等交互与此前一致；展开入口为纯 chevron。
   - 已选状态：卡片边框/底色/弱 ring。

八、语言规范
1) 项目/专业名称：英文优先展示（nameEn）。
2) 中文用于导航、状态、辅助操作文案。

九、技术与存储
1) 本地存储键
   - edumatch_questionnaire
   - edumatch_match_result
   - edumatch_added_programs
   - edumatch_application_status
2) 组件规范
   - 优先使用 Vercel/shadcn 组件（已开始替换高频输入控件）。

十二、2026-04-09 对话需求摘要（归档）
本次对话中已落实到文档与实现的约定如下：
1) 匹配页一屏信息密度：大屏双栏独立滚动、压缩顶栏与工具区、去掉重复统计与分栏标题，减少整页滚动。
2) 简化同页不必要信息：去掉面包屑式页面标题、空状态大图标、与列表重复的概览数字等（以当前代码为准）。
3) 学制 / 截止 / 费用：界面只用图标 + 数字或金额字符串，不写中文标签；学校侧排名与项目数同理。
4) 匹配说明：不写申请批次建议；首条说明紧扣问卷中教育经历与目标方向等已填字段。
5) 卡片视觉：圆角 xl、轻阴影、选中态弱 ring；学校卡与项目卡风格统一。

（说明：修改匹配规则或文案后，用户需「重匹配」或清空 edumatch_match_result 后重新生成，才能看到最新匹配原因。）
